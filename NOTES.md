# Work Plan & Session Log

## Framework Goal

A React meta-framework built on a **template-and-slots** architecture (not nested layouts) with **automatic interactivity inference** (not `"use client"` directives). Ships zero JavaScript for static content and minimal JavaScript per interactive slot. The codegen layer is **pure** (no filesystem, no Vite deps) so it can run inside Cloudflare Workers for multi-tenant dynamic bundling. The runtime is **platform-agnostic** (Workers, Bun, Node). Designed for AI-driven, token-efficient multi-tenant apps on the edge.

## Plan

### Phase 1: Functional Testing (fix everything that's broken)

1. **Build the monorepo** — `pnpm install && pnpm build`. Fix any build errors.
2. **TypeScript check** — `pnpm check-types`. Fix type errors across all packages.
3. **Run existing tests** — `pnpm test`. Document pass/fail state.
4. **Test slot types in playground**:
   - Fully static slots (zero JS shipped)
   - Partially static/hydrated slots (some slots interactive, some not)
   - Fully hydrated slots (all slots interactive)
   - Fallback slots (slot not filled by page, template provides fallback)
   - Required slots (missing required slot = build/runtime error)
5. **Test HMR** — Start Vite dev server, edit route files, verify hot reload works for both static and hydrated slots.
6. **Test codegen purity** — Verify `@zerohuman/framework/codegen` has zero Node.js/Vite imports. Must be runnable in a Cloudflare Worker.
7. **Test Vite plugin** — Verify `@zerohuman/framework/vite` starts, generates types, transforms routes.
8. **Test generated files** — `framework.gen.d.ts` and `routes.gen.ts` generate correctly via Vite dev and CLI.
9. **Test production build** — `vite build`, serve the output, verify SSR + hydration work in prod mode.
10. **Test error boundaries** — Loader throws, slot errors, missing required slots, 404/500 error pages.
11. **Test edge cases** — Nested routes, dynamic params (`[slug]`), deeply nested params (`[id]/posts/[postId]`), catch-all routes.
12. **Test types** — Context types, loader data types, error context types are intuitive and correctly inferred.

### Phase 2: Cloudflare Dynamic Workers (prototype branch)

13. **Check prototype branch** — Understand how `@zerohuman/framework/codegen` + `@cloudflare/worker-bundler` work together.
14. **Verify codegen runs in Wrangler** — Build a minimal Worker that imports codegen and generates routes at runtime.
15. **Test dynamic tenant worker hosting** — End-to-end: codegen in Worker → bundle tenant app → serve.

### Phase 3: Integration Tests

16. **Write integration tests** for each slot type (static, partial, hydrated, fallback, required).
17. **Write integration tests** for error pages (404, 500, loader errors, error page errors / circuit breaker).
18. **Write integration tests** for codegen output (routes.gen.ts shape, framework.gen.d.ts shape, hydration manifest).
19. **Write integration tests** for production build output (correct assets, correct manifest, zero JS for static pages).
20. **Write integration tests** for module boundaries (runtime doesn't import codegen/vite, codegen doesn't import vite).

### Phase 4: API Improvements (OSS-friendly)

21. **Package export validation** — Verify `package.json` exports map, ensure consumers can import all public APIs cleanly. Inspired by Vite's clean export map.
22. **Dependency audit** — No circular deps between packages, correct peer deps. Inspired by TanStack's strict dep management.
23. **Error messages** — Clear, actionable error messages with suggestions (like Vite's "did you mean..." errors, Astro's error overlay).
24. **API naming review** — Ensure all public API names are intuitive. Compare with Next.js/Remix/Astro naming conventions.
25. **Bundle size check** — Framework bundle must stay small for the codegen-on-Worker use case.

### Phase 5: Code Quality (AAA standard)

26. **JSDoc on all exported functions and types** — Single-line, per ADR-0009 code style rules.
27. **File ordering** — imports → types → constants → private functions → exported functions (per ADR-0009).
28. **Blank lines between logical steps** — Each phase in a function body gets visual separation (per ADR-0009).
29. **Re-exports only in barrel files** — No `export {}` blocks in internal modules (per ADR-0009).
30. **Readable grouping and spacing** — Well-structured files that read like prose.

### Phase 6: ESLint & Tooling

31. **Review ESLint config** — Either make it stricter (catch real bugs) or less strict (remove noise). Inspired by Astro's pragmatic ESLint config.
32. **Prettier config review** — Ensure consistency.
33. **CI pipeline** — Integration tests should run in CI if not already wired.

### Phase 7: Continuous Improvement

34. **Changelog / migration notes** — Document any breaking API changes.
35. **README** — Root README explaining architecture (if going OSS).
36. **Keep improving** — Each loop picks the highest-impact remaining work.
